Temple Core Module
 

About

This is version 3.2.0 of the Temple Core Module, built on 2012-10-11. Get the latest version from: http://code.google.com/p/templelibrary/. You can get the files from SVN or you could download it as a zip from the downloads page.

This module contains the core of the Temple. All other Temple modules are based on this core. The core adds some basic functionality to the native Flash classes which are used in the Temple. The core module handles to following functionality:

  • Destruction
  • Advanced Event model
  • Logging
  • Memory Management
  • Debugging
  • Some extra properties for DisplayObjects

Destruction

All Temple objects implement the IDestructible interface which adds the destruct() method to every object. Calling this destruct method will destroy the object, removes all its event listeners and makes it available for garbage collection.

Advanced Event model

All Temple's EventDispatchers implement the ICoreEventDispatcher interface. This interface adds to following functionality to the object:

  • addEventListenerOnce()
  • removeAllEventListeners()
  • removeAllOnceEventListenersForType()
  • removeAllStrongEventListenersForType()

Logging

Every Temple object has some quick methods for sending log messages to your favorite log application. There is a method for every level of logging:

  • logDebug()
  • logInfo()
  • logWarn()
  • logError()
  • logFatal()
  • logStatus()

Note that these logging calls are protected and can only be called from inside to (extended) object. But you can always use the related methods of the Log class.

For viewing these log messages you could simply use the trace panel in the Flash IDE. Or you could listen for LogEvents on the Log class, with the addLogListener() method. You could also use Yalala to view you log message. You need the Yalog module to implement Yalala in your project.

Memory Management

The destruct model and the advanced event model of the Temple can help you to get rid of your old objects. However it is still possible that these objects aren't garbage collected. Holding a reference to a destructed object can block the Garbage Collector for disposing the object. By using the Memory class you can track down these objects. If you set Temple.registerObjectsInMemory to true all Temple objects are registered in the Memory, using a weak reference. If a destructed object can still be found in the Memory, you might have a memory leak.

Debugging

Many Temple classes implement the IDebuggable interface. This interface adds a debug property to the object. Setting this property to true enables debug mode for this object. Depending on the implementation of the object, extra debug information will be available. Most objects will send extra debug messages which can help you to debug your application.

This debug property can be set manually on every object you want to debug, or you could use the DebugManager to set the debug property for every object at once. If you set DebugManager.debugMode to DebugMode.ALL you enable debugging for every IDebuggable which is added in the DebugManager.

The objects in the Temple are added to the DebugManager with the addToDebugManager() function. This function only adds the Object if the DebugManager is available and doesn't force the compiler to add the DebugManager in the SWF. Therefore you MUST use the DebugManager in your own code to force the compiler to add the DebugManager into the SWF.

The DebugManager checks for the property debug in the URL. If this property is found it will set the DebugManager to this debugMode. In this way it's possible to set the debugMode on a live website by just adding "?debug=all" to the URL: 

http://www.some-domain.com/?debug=all

Extra properties for DisplayObjects

All DisplayObjects of the Temple have some extra properties, which can be quite handy:

  • autoAlpha
  • position
  • scale
  • stage
  • children (only for DisplayObjectContainers)

Content

This module contains the following folders:

/bin
Contains the SWC of this module.
/doc
Contains the documentation of the core module.
/examples
Contains some examples of the core module.
/export
Contains the generated source files from the /source folder.
(This folder is not available in the zip file.)
/source
Contains the source (.as) files of this module, but with include statements.
Note: the source folder of the zip file contains the generated (exported) source files without include statements.
/tools
Contains ANT scripts which are used to generate docs, examples, swc etc...
(This folder is not available in the zip file.)

Dependencies

The core module has no dependencies with other modules and can be used straight away.

Getting started

Adding the core module to your project

To add the core module to your project you could use the SWC file or the AS files. The SWC file can be found in the /bin folder. If you want to use the SWC file you need to add the SWC file to the class path of your project.

If you want to use the AS files you just need to copy the content of the /export folder to the source folder of your project. Or you could add the /export folder to the class path of your project. It's also possible to use the AS files in the /source folder, but since these files use the include statement, your code editor might not understand this.
NOTE: don't copy folders from an SVN check out, since this can cause SVN problems. It would be better to do an SVN export to your own folder.

Use the Temple features in your own classes

To use the Temple features in your own classes you only need to extend the classes of the Temple. Temple has a core class for almost every native Flash class. Instead of extending MovieClip you extend CoreMovieClip, or CoreSprite instead of Sprite, CoreTimer instead of Timer or CoreEventDispatcher instead of EventDispatcher, etcetera, etcetera.

Examples

Check out the /examples folder for all examples. This folder also contains a list of all available examples.

Note: every example has an XML file which is an ANT build file which can be used to build the example.

Notes

Source and export folder

As you may notice there is a source folder and an export folder in the core module. The source folder contains the actual source files. But these files use the include statement. Not every code editor knows how to handle this include statement. Therefore we have added the export folder which contains the same code, but we replaced the include with the content of the actual include file. This replacement is done using ANT. These ANT scripts can be found in the tools/ant/folder
